# DBTT Car Modification — Vos Automotive Digital Platform

A digital customization platform for **Vos Automotive (Singapore)** that lets customers visualise car modifications in 3D, book appointments, and provides owners and workers with a live analytics dashboard.

The project is split into two parts that must both be running at the same time:

| Part | Technology | Port |
|---|---|---|
| **Frontend** | React + Vite + TypeScript | `http://localhost:5173` |
| **Backend** | Python + FastAPI | `http://localhost:8000` |

---

## Setup & Running

You need **two separate terminals** open at the same time — one for the backend, one for the frontend.

---

### Terminal 1 — Backend (Python / FastAPI)

```bash
# 1. Navigate to the backend folder
cd backend

# 2. (Recommended) Create and activate a virtual environment
python -m venv venv

# On Windows (Git Bash) — use this:
source venv/Scripts/activate

# On Windows (Command Prompt):
# venv\Scripts\activate.bat

# On Windows (PowerShell):
# venv\Scripts\Activate.ps1

# On macOS/Linux:
# source venv/bin/activate

# 3. Install Python dependencies
pip install -r requirements.txt

# 4. Start the backend server
python -m uvicorn main:app --reload
```

> **Note:** Use `python -m uvicorn` instead of just `uvicorn` — this avoids PATH issues on Windows where the command may not be recognised even after installing.

The backend is ready when you see:
```
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
```

You can verify it is working by opening `http://localhost:8000/api/kpis` in your browser — it should return JSON.

---

### Terminal 2 — Frontend (React / Vite)

```bash
# 1. Navigate to the frontend folder
cd frontend

# 2. Install Node dependencies (only needed once, or after pulling changes)
npm install

# 3. Start the development server
npm run dev
```

The frontend is ready when you see:
```
  VITE v6.x.x  ready in Xms

  ➜  Local:   http://localhost:5173/
```

Open `http://localhost:5173` in your browser.

## Important: Both servers must be running

The Owner Dashboard fetches all analytics data from the backend. If the backend is not running, the owner dashboard will show blank/loading panels. The rest of the app (customizer, booking, worker jobs) works independently.

## Backend — How It Works

The backend folder contains three Python files. Here is what each one does.

### `main.py` — API Entry Point

`main.py` is the FastAPI application that the server runs. It does three things:

1. Creates the FastAPI app and configures CORS so the React frontend (port 5173) is allowed to call it.
2. Imports functions from `analysis.py` and `mock_data.py`.
3. Registers each function as a GET endpoint (e.g. `GET /api/projections` calls `get_projections()` from `analysis.py`).

Nothing is computed inside `main.py` itself — it is purely the routing layer that connects the HTTP API to the business logic in the other two files.

---

### `analysis.py` — Notebook Logic as Python Functions

`analysis.py` is a direct port of the business analysis notebook (`analysis.ipynb`, sections 4–11) into callable Python functions. The frontend owner dashboard fetches its analytical charts from these functions via the API.

It contains four main functions:

| Function | Notebook Section | What it produces |
|---|---|---|
| `get_projections()` | Sections 4–7 | Synthetic historical data (2020–2024) plus a 5-year forecast (2025–2029) under two scenarios: **Baseline** (6% organic growth) and **Digital** (21% growth with conversion and upsell lifts applied). Returns both scenarios and a year-by-year comparison showing the revenue and customer uplift from going digital. |
| `get_service_stats()` | Section 9 | Generates a synthetic dataset of 320 customer transactions (using a fixed random seed for reproducibility), then computes: most popular services by mention count, top service bundles purchased together, average order value per service, customer distribution across budget bands, and style preference breakdown. |
| `get_recommendations()` | Section 10 | Runs association rule mining on the same 320-customer transaction dataset. For each service, it finds the service most frequently purchased alongside it and returns the confidence score — used to power the cross-sell recommendation table on the owner dashboard. |
| `get_uplift_kpis()` | Section 11 | Aggregates the outputs of the three functions above into a set of headline KPI findings: total 5-year customer and revenue uplift, average annual uplift, top bundle, top service, and the strongest cross-sell rule. |

All data is synthetic and generated deterministically from fixed assumptions (defined in `_ASSUMPTIONS`) and a seeded random number generator (`np.random.default_rng(42)`), so results are identical on every run.

---

### `mock_data.py` — Synthetic Operational Data

`mock_data.py` provides data for dashboard items that have no direct equivalent in the analysis notebook — operational figures that a business would track day-to-day rather than derive from a projection model.

| Function | Endpoint | What it provides |
|---|---|---|
| `get_kpi_stats()` | `/api/kpis` | Four KPI card values: total revenue, orders this month, average order value, and top service. All figures are consistent with the notebook's 2025 digital scenario. |
| `get_monthly_revenue()` | `/api/monthly-revenue` | Monthly revenue breakdown for 2025 across 12 months. The 12 values sum exactly to SGD 525,609, matching the notebook's 2025 digital revenue projection. |
| `get_service_breakdown()` | `/api/service-breakdown` | Revenue contribution per service category (Car Wrap, PPF, Bodykit, etc.), derived from customer counts and per-service average order values from the notebook's transaction analytics. |
| `get_trend_insights()` | `/api/trend-insights` | A list of modification and finish trend items (e.g. "Widebody Kit up 41%"), based on the synthetic order distribution. Used to populate the trend insights panel on the owner dashboard. |
| `get_supply_alerts()` | `/api/supply-alerts` | Inventory items at or below minimum stock levels, flagged as `warning` or `critical`. Ported from the frontend mock inventory data into Python so all data originates from one place (the backend). |

## Optional: AI Chatbot (Customizer)

The AI chatbot on the customizer page requires an OpenAI API key. Without it the chatbot will not respond, but the rest of the app works normally.

Create a `.env` file inside the `frontend/` folder:

```
VITE_OPENAI_API_KEY=sk-...your-key-here...
```
